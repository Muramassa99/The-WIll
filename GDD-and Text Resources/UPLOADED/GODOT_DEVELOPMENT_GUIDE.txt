PROJECT SCOPE SUMMARY
=====================
A seasonal, social dungeon crawler featuring:
- Central hub (The City) for player gathering & reputation building
- Instanced procedural floors (15-60min expeditions)
- Weapon crafting from harvested materials (block-based system)
- Phase-based ability system tied to equipment
- Server-authoritative combat with knowledge/skill progression
- Modding support via Thunderstore
- Player-hosted or dedicated server architecture

GODOT SUITABILITY: ✓ EXCELLENT FIT


GODOT DOCUMENTATION RESEARCH LAW
================================

Before implementing any Godot code change, do topic-specific online documentation research first.
This applies even to trivial engine tasks.

Authority order for Godot work:
1. Official Godot documentation for the active engine branch/version.
2. Official documented native tools and workflows combined where needed.
3. Custom project-side solutions only after confirming the official documented path does not fully cover the need.

Practical rule:
- Do not start from chat memory when Godot behavior or API details matter.
- Look up the exact Godot topic first.
- Then decide whether native tools already solve it.
- Only after that should custom implementation begin.

=== PHASE 1: FOUNDATION (Weeks 1-8) ===

1.1 CORE SYSTEMS TO BUILD FIRST
================================

A) Player & Combat Foundation
   - 3D character controller (OTS camera, movement, dodge)
   - Hit detection & damage resolution
   - Status effect system (poison, slow, stagger, shield)
   - Durability tracking

   Godot Components:
   - CharacterBody3D for movement
   - RayCasts/Areas for hit detection
   - Signal system for damage events
   - Custom StateMachine for character states

B) Weapon Crafting & Profile System
   - Block-based weapon builder (grid system)
   - Material definitions (density, hardness, elemental affinity)
   - Weapon profile generator (mass, reach, balance, flex)
   - Profile → Ability mapping system

   Godot Components:
   - Custom Resource classes for weapon blocks
   - Crafting calculator (shape + materials → stats)
   - Ability factory that builds from templates
   - Items system with durability

C) Ability System (Phase-based)
   - Windup → Active → Recovery phases
   - Hit volume system (cone/box/sweep/projectile)
   - Cooldown management
   - Iframe windows & stagger resistance

   Godot Components:
   - Ability base class with phase manager
   - Hitbox/hurtbox system
   - Cooldown tracker
   - Animation orchestration (can use simple primitives initially)

D) Simple Enemy AI
   - Pattern-based attacks
   - Consistent behavior (not RNG-heavy)
   - Scaling via numbers only (HP, damage, defense)
   - Learning-friendly patterns

   Godot Components:
   - Enemy behavior tree or state machine
   - Customizable attack patterns
   - Aggro/detection systems
   - Death/loot drop handlers

E) First Vertical Slice
   - Connect all above into 1 playable room
   - 1 basic enemy with 2-3 phases
   - Ability slots (4 fixed roles: Primary/Secondary/Mobility/Defense)
   - Loot drop (2-3 materials)
   - Simple craft at NPC
   - Equipped weapon changes available abilities

   Godot Components:
   - UI for ability slots
   - Inventory system
   - NPC interaction handler
   - Craft UI with preview

1.3 CORE SYSTEMS - IMPLEMENTATION ORDER
========================================

Step 1: Define Core Data Models (2-3 days)
  - Create Material, Block, ItemDefinition, AbilityTemplate Resources
  - Weapon profile calculator
  - Loot table definitions
  
Step 2: Build Stat System (3-4 days)
  - Base stats: HP, Energy, Armor, Precision, Evasion, Magic Resistance
  - Modifier application (gear + buffs)
  - Scaling formulas
  
Step 3: Implement Combat Resolver (4-5 days)
  - CombatResolver class (only source of truth for damage calc)
  - Hit detection logic
  - Mitigation calculations
  - Durability loss tracking
  
Step 4: Build Ability System (5-7 days)
  - Ability base class with phase system
  - Cooldown manager
  - Hitbox/hurtbox implementation
  - Iframe window handler
  
Step 5: Crafting System (4-5 days)
  - Block grid editor (in-game or editor-based initially)
  - Shape → Profile converter
  - Material + shape → Stats calculator
  - Weapon instance creation

Step 6: Loot & Economy (2-3 days)
  - Drop resolver
  - Inventory system
  - Item durability & repair
  
Step 7: Player Movement & Camera (3-4 days)
  - OTS camera system
  - Character movement controller
  - Dodge mechanic
  - Collision handling
  
Step 8: First Enemy (2-3 days)
  - Basic patrol/aggro
  - Simple attack pattern (2-3 phase moves)
  - Death & loot drop
  
Step 9: Connect All (3-4 days)
  - First playable room
  - Player spawning
  - NPC interaction (craft UI)
  - Re-equip weapons & see ability changes


=== PHASE 2: MULTIPLAYER FOUNDATION (Weeks 9-14) ===

2.1 NETWORKING ARCHITECTURE
=============================

Decision: Server-Authoritative Model
  - Client sends inputs (move, cast, interact)
  - Server runs all core systems
  - Server sends results (position, hit, damage)
  - Anti-cheat validation on server

Godot Multiplayer Setup:
  - Use built-in MultiplayerSynchronizer for position/animation replication
  - Custom authority layer for combat/crafting validation
  - Player-hosted: One player runs server instance
  - Dedicated: Can run headless Godot server

2.2 PLAYER SESSION SYSTEM
  - Account data (character, inventory, progression)
  - Session persistence (save/load between sessions)
  - Character switching (optional for later)

2.3 FLOOR INSTANTIATION
  - Seed-based procedural generation
  - Player can enter → creates floor instance
  - Floor state managed by authority player or server
  - 4-player capacity per instance

2.4 PARTY SYSTEM
  - Hub interaction creating parties
  - Shared floor instances for party members
  - Loot distribution (bid/pass system)

=== CRITICAL DESIGN DECISIONS ===

Combat Authority
  - Server ALWAYS calculates damage
  - Server validates hit timing
  - Client cannot modify cooldowns
  → Prevents speedhack + RMT dominance

Crafting Authority
  - Server validates block layout + materials
  - Server generates stats from profile
  - Client UI is preview only
  → Prevents item duplication/illegal stats

Movement Authority
  - Server checks if movement is physically valid
  - Server validates ability cast legality
  - Snapshots sent to other players
  → Prevents teleport hacks

Loot Distribution
  - Server rolls rarity & item type
  - Players bid/pass → server processes results
  - Inventory modified only by server
  → Prevents loot theft in multiplayer


=== GODOT-SPECIFIC IMPLEMENTATION NOTES ===

3D vs 2D
  - Recommended: Node3D with orthographic camera for "isometric-like" 3D feel
  - Allows real 3D depth without full 3D complexity
  - Can mix 2D UI over 3D world easily

Performance Optimization
  - Use spatial partitioning for enemies (Decals hidden if >500m away)
  - Pool projectiles & VFX
  - Disable physics for far players in multiplayer
  - Instance LOD system for distant enemies

Modding Hooks
  - Use signals extensively (OnCraft, OnCombat, OnLevelLoad)
  - Allow scripts in res://mods/ folder
  - Data-driven ability system (no hardcoded skills)
  - Color tags system for items (extensible)

Save System
  - Use Resource system for save data
  - JSON export for backup/sharing seeds
  - Version migration layer for updates
