## The naming law

This is the template I want you to follow so names do not drift.

* `*Def` = static definition data, usually a `Resource`
* `*Atom` = smallest stored unit
* `*State` = mutable runtime state
* `*Instance` = a concrete owned thing in play/in inventory
* `*Profile` = baked/derived gameplay-facing result
* `*Resolver` = pure rule logic that converts one layer into another
* `*Service` = orchestration / application layer, no scene ownership
* `*Controller` = scene/node behavior
* `*Registry` = lookup table / index
* `*Factory` = builds legal runtime objects from definitions/profiles

That matches your atomic-first stat/material setup, where `StatLine` is the universal smallest stat unit and bigger systems are built upward from stored atoms.

## What each floor of the project does

### `core/`

This is the law of the machine.

No scene ownership. No UI assumptions. No вЂњgrab this node and do magic.вЂќ
This is where the atomic-first logic lives: `StatLine`, material/tier definitions, WIP/baked item models, and the resolvers that turn one layer into another. That follows both your atomic math sheet and the Forge rule that gameplay should stop reading raw cells once a baked profile exists.  

### `services/`

This is orchestration.

A service is allowed to say:

* take these drops
* process them into forge mats
* save that WIP object
* bake this test print
* finalize this item

But it should **not** own visual/editor behavior. It coordinates rules and persistence.

### `runtime/`

This is the node world.

These controllers are the Godot-facing layer:

* player movement
* forge viewport behavior
* cameras
* hitboxes
* dummies
* enemy AI nodes
* station interaction

This is where you вЂњuse Godot,вЂќ not where you define the truth of the systems.

### `scenes/`

Only composition.
Scenes should wire together controllers and visuals, not become secret logic warehouses.

### `data/`

Actual `.tres` and content defs.
This is where you store tiers, materials, thresholds, ability defs, and later enemy defs.

### `saves/`

Do **not** merge save domains too early.
You already know WIP forge objects, finalized items, and profile/world data are different lifecycles. Keep them separate from the start.  

## The dependency law

This is the part that matters most.

### Allowed direction

`defs/atoms -> models -> resolvers -> services -> runtime controllers -> scenes/ui`

### Not allowed

* UI reading raw forge cells directly for gameplay truth
* runtime controllers doing combat math locally вЂњbecause it was convenientвЂќ
* finalized combat reading `CellAtom[]` when a `BakedProfile` exists
* forge tools writing directly into player combat state without going through services/models

That is the same hidden structure you wanted: visible freedom for the player, strict resolver structure underneath.

## The naming and save split you must not violate

There are four objects that must **never** collapse into one type:

* `ForgeMaterialStack`
* `CraftedItemWIP`
* `TestPrintInstance`
* `FinalizedItemInstance`

If you make one вЂњitem.gdвЂќ and try to force all four meanings into it, that is how the spaghetti starts.

Why this matters:

* processed forge mats are not world drops
* WIP objects are not finalized items
* test prints are not tradeable exports
* finalized items carry provenance, trade history, and later blueprint/salvage implications

Your own notes already require different lifecycle laws for these.  

## My one structural correction to the earlier Godot guide

The earlier guide had a solid direction, but I would make one change for your project:

Do **not** let `runtime/player/equipment.gd` or `runtime/player/abilities.gd` become the place where stat math or capability logic lives.

Those controllers should ask the core/services layer for answers.

So instead of:

* player node computes attack/defense/crit/capability logic

Do:

* player node asks `equipment_service`
* service asks `stat_aggregation_resolver`
* service asks `profile_resolver` / `behavior_resolver`
* runtime only applies the result

That keeps your reverse-engineered final machine intact.
