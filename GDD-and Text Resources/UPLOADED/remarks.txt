=== ARCHITECTURAL WINS (Things You Got Right) ===

1. PIVOT TO PLAYER-HOSTED MODEL
   Why This Matters:
   - Eliminated global economy policing (massive engineering cost)
   - Removed RMT enforcement complexity
   - Made modding first-class instead of afterthought
   - Puts server burden on community (Minecraft/Valheim model)
   
   Impact:
   - You can ship without anti-cheat team
   - Server authority still prevents client-side cheating
   - Modding = content pipeline = free development
   
   Implementation Note:
   - Godot supports peer-to-peer natively
   - One player as authority per floor instance
   - Can optionally offer official "dedicated servers" later


2. SEPARATION OF PLATFORMS (Local vs Multiplayer)
   Why This Matters:
   - Allows solo testing without networking complexity
   - Same core systems work offline and online
   - Easier pathfinding: build offline first, add MP later
   
   Implementation Note:
   - Write core systems (combat, crafting) to not depend on Multiplayer API
   - Wrap MP calls in conditional layer
   - Test singleplayer first, MP becomes "networking wrapper"


3. KNOWLEDGE ECONOMY INSTEAD OF GEAR TREADMILL
   Why This Matters:
   - Allows mod-created content to be viable
   - Veterans have advantage without gatekeeping progression
   - Modded floors/bosses stay competitive with official content
   - No power-creep arms race
   
   Implementation Note:
   - Bosses should be pattern-based (learnable)
   - Difficulty comes from knowledge/optimization, not stats
   - Loot should plateau (vertical progression slows end-game)
   - Modding adds content breadth, not permanent power increase


4. SEED-BASED GENERATION
   Why This Matters:
   - Reproducible worlds (essential for bug fixing)
   - Shareable content (Minecraft-style codes)
   - No server-side storage of floor layouts
   - Community content distribution (Thunderstore)
   
   Implementation Note:
   - Use FastNoise2D or Godot's built-in noise
   - Seed both layout AND enemy spawns
   - Document seed structure clearly for modders
   - Test: Same seed on different continents = same dungeon


=== DESIGN GAPS & WARNINGS ===

1. ABILITY BALANCE ACROSS WEAPON TYPES
   Problem:
   - You can craft infinite weapon variations
   - Each variation needs different base ability stats
   - Risk: Some shapes become objectively superior
   - Risk: Crafting becomes "find optimal grid" instead of creative
   
   Mitigation:
   - Set hard constraints on weapon profiles (max mass, reach ranges)
   - Use block limits (max 20 blocks per weapon)
   - Make multiple solutions equally viable (speed vs power tradeoff)
   - Playtestable: Every viable shape should work against all bosses
   
   Recommendation:
   - Implement weapon profile validator
   - Create balanced reference weapons first
   - Test that crafted weapons don't exceed reference power
   - Make balance transparent (show stats before craft)


2. MODDING SCOPE EXPLOSION
   Problem:
   - "Modding keeps content alive" is true
   - But modding infrastructure is a product responsibility
   - You still need to validate mods don't break economy
   - You still need deprecation support when you change systems
   
   Mitigation:
   - Start with read-only modding (add content, not change core)
   - Mod API versioning from day 1
   - Clear mod compatibility layers
   - Don't promise "modders can do anything" until proven
   
   Recommendation:
   - Phase 1: Official content only
   - Phase 2: Modding API (restricted scope)
   - Phase 3: Community marketplace (after you understand demand)
   - Use Thunderstore config strictly (version locks)


3. MULTIPLAYER SYNC COMPLEXITY
   Problem:
   - Server authority is right decision
   - But syncing 4 players + enemies + hitboxes is hard
   - Godot MP can lag without careful optimization
   - Player-hosted authority creates latency variance
   
   Mitigation:
   - Use client-side prediction for movement only
   - Server reconciliation on combat (position must be validated)
   - Hitbox validation: Server checks, not client
   - Test high-latency scenarios (client on 250ms ping)
   
   Recommendation:
   - Build dedicated headless server artifact
   - Offer free official servers (or Thunderstore official hosts)
   - Allow peer hosting but recommend dedicated
   - Test limit: 4 players per instance max


4. SEASONAL RESET SCOPE
   Problem:
   - "Floors reshuffle" means procedural generation must be consistent
   - "Leaderboards reset" implies tracking across seasons
   - "Progress remains" implies character progression persists
   - These create save versioning complexity
   
   Mitigation:
   - Define "progress = character level/gear", "memory = boss clears"
   - Save to separate files (progress.tres vs memory.tres)
   - Test: Season switch loads old progress, clears memory
   - Track by character UID + timestamp
   
   Recommendation:
   - Implement this in Phase 2, not Phase 1
   - Create season manager class (handles rotation + resets)
   - Test manual season change test early


=== TECHNICAL DEBT WARNINGS ===

1. ANIMATION & ART PIPELINE
   What You Don't Have Yet:
   - No character animations
   - No enemy animations
   - No VFX system
   - No audio design
   
   Impact:
   - Game will feel unfinished without polish
   - Can use programmer art (colored cubes) for MVP
   - Animation dependencies: artist, rig setup, state machine per enemy
   
   Recommendation:

   - Phase 2: Simple skeletal rigs + basic animations
   - Phase 3: Polish animations, add VFX

2. CAMERA & VISUAL CLARITY
   What Matters:
   - OTS camera is hard to get right
   - Need clear visual feedback for hits
   - Need readable text (UI scales with 4K testing)
   - Need depth clarity (which enemy does attack hit?)
   
   Recommendation:
   - Test camera early with debug shapes
   - Create clear visual feedback (screen shake, color flash on hit)
   - Log all hits to console (can't see everything visually)
   - Record gameplay on hardware you DON'T own (borrow to test)


3. SAVE SYSTEM COMPLEXITY
   What You're Building:
   - Per-character saves
   - Per-season progress
   - Shared world seed/leaderboard data
   - Mod compatibility versioning
   
   Recommendation:
   - Use Godot's Resource system (.tres files)
   - Version saves from day 1 (save format = v1.3)
   - Test: Old save loads in new version (migration script)
   - Backup logic: Keep last 5 saves


4. PROGRESSION PACING
   Problem:
   - What does end-game look like?
   - When do players stop logging in?
   - How does new mod content fit progression?
   
   Recommendation:
   - Define vertical progression ceiling (gear tiers)
   - Define horizontal progression (difficulty tiers + specialty builds)
   - Seasons should add new challenges, not mandatory gear
   - Allow veteran players to help newcomers (co-op benefits both)


=== MODDING & CONTENT SPECIFICS ===

What Modders CAN Change (Safe):
  ✓ Enemy patterns (new attack patterns)
  ✓ Asset overrides (custom 3D models, textures, sounds)
  ✓ Ability cosmetics (VFX, sounds for abilities)
  ✓ New floors (seeded procedural layouts)
  ✓ New material types (add to crafting pool)
  ✓ New enemy types (with balanced stats)
  ✓ Quest/narrative content
  ✓ UI themes & skins

What Modders CANNOT Change (Locked):
  ✗ Combat resolver (damage calculation must be authoritative)
  ✗ Stat formulas (must be server-side only)
  ✗ Drop tables (must be server cryptographic to prevent RNG hacking)
  ✗ Character progression (level/XP) must be server-validated
  ✗ Inventory ownership (must be account-based)
  ✗ Season schedules (official only)

Implementation Note:
  - Document these boundaries early
  - Provide mod templates for safe changes
  - Use Godot's plugin system if possible
  - Consider hot-reload for modding (fast iteration)

=== NETWORKING MODEL SPECIFICS ===

Player-Hosted Authority Flow:
  
  Player A (Host): Runs server instance
    → Owns floor state
    → Calculates damage
    → Determines loot rolls
    → Sends snapshots to Players B, C, D
  
  Players B, C, D (Clients):
    → Send input: move direction + ability cast
    → Receive snapshots: positions, animations, status
    → Run animation/prediction locally
    → Authority verifies legality before committing
  
  Implications:
    - Host advantage: ~0ms latency, sees everything
    - Clients: Lag if host's connection bad
    - Mitigation: Auto-rebalance host (vote if host lags)
    - Client-side prediction: Smooth movement despite lag
    - Authority validation: Catch cheaters post-hoc

  Recommendation for 4-Player Limit:
    - More players = more bandwidth needed
    - More players = longer combat calculations
    - Test with 4 simultaneous players + 8 enemies early
    - If performance OK, can expand to 8 later

=== WHAT COULD GO WRONG (And What Doesn't) ===

Most Likely Problem #1: Animation Pipeline Didn't Scale
  - You built 1 enemy animation, but 8 enemies need all actions
  - Takes 10x longer than expected
  - Solution: Use same skeletal rig, only change controller logic
  - Fallback: Godot IK system can improvise animations

Most Likely Problem #2: Procedural Generation Looked Boring
  - Seed 1 looks like seed 2 like seed 3
  - Players feel: "Why explore if all dungeons same?"
  - Solution: Layered generation (base structure + variation passes)
  - Test early with visual debug (color-code biomes, treasures)

Most Likely Problem #3: Networking Desync
  - Player A sees enemy hit, Player B doesn't
  - Loot distribution roles back after players leave
  - Solution: UnitTest the sync logic before multiplayer
  - Use deterministic tick system (server controls frame timing)

Most Likely Problem #4: Balance Required Mid-Development
  - Boss too easy/hard
  - Weapon X objectively better than Y
  - Solution: Build tuning spreadsheet early, test weekly
  - Implement admin commands (!difficulty, !loot_chance) early

Doesn't Go Wrong (Usually):
  ✓ Core combat loop (you have clear design)
  ✓ Crafting system (logic is deterministic)
  ✓ Single-player MVP (no MP chaos)
  ✓ Godot project structure (clear separation)
