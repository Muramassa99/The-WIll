And then one bonus pack that I think you actually need too:

That matches the split we already locked: material says what matter can contribute; context/intent says what kind of thing the player is trying to build; profile says what it actually became. 

---

# 0) The law before the list

Use this split:

* `base_stat_lines` = direct numeric stats that can end up affecting final gear numbers
* `capability_bias_lines` = hidden “what this matter supports physically/behaviorally”
* `skill_family_bias_lines` = hidden “what kind of skill families this matter leans toward”
* `elemental_affinity_lines` = hidden “what elements this matter brings/supports”
* `equipment_context_bias_lines` = hidden “what kind of craft this matter makes sense in”
* `forge_intent_ids` = player/forge-side starting bias, not material truth

That split fits both your atomic-first math sheet and your crafting-atoms doc.

```text
cap_anchor
cap_aoe
cap_counter
cap_parry
```

## What these mean, briefly

* `cap_edge` = supports cutting/bladed behavior
* `cap_blunt` = supports impact/crushing behavior
* `cap_pierce` = supports thrust/penetration
* `cap_guard` = supports blocking/protective behavior
* `cap_flex` = supports bend/whip/string behavior
* `cap_tether` = supports leash/rope/chain behavior
* `cap_launch` = supports ranged launch behavior
* `cap_magic` = supports magical behaviors/casting overlays
* `cap_heal` = supports healing/support payloads
* `cap_stability` = supports reliable handling / less chaotic movement
* `cap_anchor` = supports grip/attachment/socket-like structure
* `cap_projectile` = supports projectile-style behavior cleanly
* `cap_counter` / `cap_parry` = supports reactive defense gameplay

These should mostly be hidden from the player or shown only as simplified bars.

# 3) FAMILY IDS

These belong in `skill_family_bias_lines`.

This is your pie-chart layer.
These are the hidden behavior-family tendencies that help narrow the skill station options.

```text

fam_defense
fam_mobility

fam_control

```

I recommend adding these too:

```text
fam_single_target
fam_burst
fam_sustain
fam_counter
fam_guard
fam_channel
fam_throw
fam_summon
fam_trap
fam_zone
fam_tether
```

## Why both capability ids and family ids?

Because they answer different questions.

Capability asks:

* “Can this built thing physically/logically support this?”

Family asks:

* “What style of skill package does this build lean toward?”

Example:

* a string may give `cap_flex`, `cap_tether`, `cap_launch`
* that may translate into `fam_ranged`, `fam_control`, `fam_mobility`

That separation will save you later.


## Practical recommendation

For code, I would keep the ids exactly in this `elem_*` format, even if display names in UI are cleaner:

* `elem_pyro` display name “Pyro”
* `elem_aqua` display name “Aqua”
* `elem_healing` display name “Healing”
* `elem_terre` display name “Terre”

That keeps data stable and UI flexible.

## Optional simplified gameplay grouping later

Under the hood you can keep the full lore names, but later build groupings like:

* fire-family = `elem_pyro`, `elem_lava`, `elem_explosion`
* death-family = `elem_death`, `elem_undeath`, `elem_plague`, `elem_corrupt`
* sky-family = `elem_caelus`, `elem_cloud`, `elem_storm`, `elem_lightning`

That way you do not lose richness.

## My recommendation

Keep it simple at first:

* use `ctx_*` for what kind of thing is being crafted
* use `intent_*` for what kind of behavior style the player is aiming for

So:

* tab = `ctx_weapon`
* sub-intent = `intent_ranged`

That is cleaner than turning one field into two jobs.

# 7) WHICH GROUP GOES WHERE

## In `BaseMaterialDef`

These arrays make sense:

### `base_stat_lines`

Use:

* direct numeric ids only

### `capability_bias_lines`

Use:

* `cap_*`

### `skill_family_bias_lines`

Use:

* `fam_*`

### `elemental_affinity_lines`

Use:

* `elem_*`

### `equipment_context_bias_lines`

Use:

* `ctx_*`

## Not in `BaseMaterialDef`

Do **not** store `intent_*` on the material as truth.
That is forge/player-side intent, not substance identity.
